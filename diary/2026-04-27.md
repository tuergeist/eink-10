- Panel is 1200Ă—825, **8 grayscale levels** (3-bit) â€” corrected from my

### Architecture decided
Three options were on the table (Arduino C++, MicroPython, server-only).
Chose: **PlatformIO firmware + PDM Python server**, with content negotiated
via a polled JSON config. The board owns nothing about layout; the server
renders a PNG and the board just displays it. Refresh skipped when the
content hash is unchanged â€” both saves the e-paper from unnecessary
rewrites and keeps battery use predictable.

Config schema:
```json
{
  "image_url": "...",
  "last_modified": "<sha256[:16] of PNG bytes>",
  "refresh_interval_seconds": 300,
  "config_url_override": null
}
```
`config_url_override` is the migration hatch â€” if we move the server to a
public host later, the existing server can hand out a new URL once and the
board persists it in NVS.

### Decisions / context worth keeping
- PDM is the user's Python tooling default â€” saved as feedback memory.
- The Inkplate library 11.0 exposes web-image methods via
  `display.image.draw(url, x, y, dither, invert)`, not as direct methods
  on `Inkplate`. The first compile attempts using `drawImage` /
  `drawPngFromWeb` failed because those live on the inner `Image` member.
- `board_build.arduino.memory_type = qio_opi` is for ESP32-S3 only; using
  it on plain ESP32 caused `sdkconfig.h not found` build failures. Removed.
- Initial venv arch mismatch: PDM picked the system python.org x86_64 build
  but installed arm64 wheels. Forced `/opt/homebrew/bin/python3` (3.14, native
  arm64) and reinstalled â€” clean.

### Mid-day pivot â€” server becomes a dumb cache
After the first end-to-end run worked but every wake re-rendered, we had two
realizations and pivoted:

1. The server-side timestamp in the rendered image meant every poll produced
   a new content hash, defeating the whole "skip refresh on no change" idea.
   Briefly fixed by removing the timestamp; deeper issue: rendering on the
   server was conceptually wrong for what we actually want.
2. The renderer should live wherever the data lives (probably wholly external).
   The server's job collapses to: **store the latest PNG, serve it under
   bearer auth, hand out `last_modified`**. Push when something changes.

Refactored the server to:
- `POST /image` (PUSH_TOKEN) with optional `?dither=floyd-steinberg` to
  quantize onto the panel's 8 grays before storing
- `GET /config.json` + `/dashboard.png` (READ_TOKEN)
- `DELETE /image` (PUSH_TOKEN) for testing
- File-backed storage at `/data/dashboard.png` + `/data/meta.json`,
  atomic writes via tempfile+rename
- Bearer tokens via env, separate push and read so a leaked board token
  can't write

Firmware updated: `Authorization: Bearer READ_TOKEN` on both calls; the
image fetch now uses `HTTPClient` with the header set, then
`drawPngFromWeb(client*, ...)` consumes the stream directly.

### Late-day iterations (post-initial-commit, not yet committed)

Clock overlay â€” settled the design as "render on the board, only when a
new image arrives, toggleable via config.json". Server change is just an
extra `overlay_clock` boolean in `/config.json` (driven by
`EINK_OVERLAY_CLOCK` env). No server-side rendering â€” keeps the cache
dumb. Firmware does NTP sync (Europe/Berlin TZ rule, `pool.ntp.org` +
`time.cloudflare.com` fallback), then draws `YYYY-MM-DD HH:MM` at
`setTextSize(2)` bottom-right with a white background and no border on
top of the freshly fetched image, before `display.display()`. Skips
overlay if config flag is false or there's no new image. So the
"timestamp" is really "last refresh time" â€” a deliberate semantic since
we don't want the board to re-render every minute (panel life + battery).

Public deploy â€” board now defaults to
`https://eink.ein-service.de/config.json` and ships the production
READ_TOKEN in the gitignored `secrets.h`. Firmware grew HTTPS support:
shared `WiFiClientSecure` with `setInsecure()` (defensible v1: bearer
token is the primary auth, TLS handles passive sniffers, certificate
pinning can be retrofitted). The `_binary_data_cert_x509_crt_bundle_*`
symbol that you'd expect from ESP-IDF is _not_ linked by the
arduino-espressif32 framework as configured here, so we sidestepped the
build-time bundle generation rabbit hole.

Currently uncommitted: firmware (HTTPS + clock overlay + smaller no-border
rendering), server (overlay_clock env), deploy (overlay_clock env in
manifest), `.env.local.example`. ~100 lines, ready for a second commit
once we know the public deploy is happy.

### Battery-aware sleep (third commit)
Inkplate 10 has a JST socket for a Li-Po and an onboard charge IC.
Without a battery the board reads ~0 V on the divider. Added:
`einkOff() + sdCardSleep()` before `esp_deep_sleep_start()` so neither
the EPD rails nor the SD card stay drawing during sleep â€” the panel was
already powered down by `display(false)` but the explicit calls are
idempotent and easier to reason about. Battery voltage is read on every
boot and logged. Below 3.4 V (and above the 2.5 V "no-battery" floor)
the firmware draws a small "LOW BAT 3.32V" overlay in the top-right of
the next freshly fetched image and multiplies the sleep interval by 6 â€”
so a 5-minute cadence becomes 30 minutes once the cell starts dipping,
buying the user time to plug in. We deliberately do **not** force a
display refresh just to show the warning; the indicator only lands when
there's a new image to draw anyway.

The library's `setPanelDeepSleep` doesn't exist on the Inkplate 10
driver class â€” the equivalent is `einkOff()`. `setPanelState()` exists
but only flips an internal flag without actually cutting power. Spent a
build cycle finding that out.
