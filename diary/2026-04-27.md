# 2026-04-27

## eink — project bootstrap

Started the e-ink dashboard project from zero.

### Hardware identified
- Detected USB-Serial bridge `1a86:7523` (CH340) on `/dev/cu.usbserial-10`.
- Auto-reset via RTS works once `esptool` sees the chip — initial connect
  failed (`No serial data received`) until the board's slide power switch /
  manual WAKE+RESET sequence got it into bootloader mode. Subsequent
  connects succeed automatically.
- Confirmed: **Inkplate 10**, ESP32-D0WDQ6 v1.0 (WROVER-E module), 4 MB
  flash (GigaDevice GD25LQ32, 1.8 V), 8 MB PSRAM, MAC `9c:9c:1f:f2:bf:3c`.
- Panel is 1200×825, **8 grayscale levels** (3-bit) — corrected from my
  initial assumption of 16 levels.

### Architecture decided
Three options were on the table (Arduino C++, MicroPython, server-only).
Chose: **PlatformIO firmware + PDM Python server**, with content negotiated
via a polled JSON config. The board owns nothing about layout; the server
renders a PNG and the board just displays it. Refresh skipped when the
content hash is unchanged — both saves the e-paper from unnecessary
rewrites and keeps battery use predictable.

Config schema:
```json
{
  "image_url": "...",
  "last_modified": "<sha256[:16] of PNG bytes>",
  "refresh_interval_seconds": 300,
  "config_url_override": null
}
```
`config_url_override` is the migration hatch — if we move the server to a
public host later, the existing server can hand out a new URL once and the
board persists it in NVS.

### What shipped today
- `server/` — PDM project, FastAPI on `:8989`, Pillow renders 1200×825
  grayscale PNG. Endpoints `/config.json`, `/dashboard.png`, `/healthz`.
  Verified: PNG decodes, hash stable across re-renders within the 5 s
  cache window.
- `firmware/` — PlatformIO targeting `esp32dev` with `-DARDUINO_INKPLATE10`.
  Wi-Fi creds in gitignored `secrets.h`. Inkplate library 11.0 from
  Soldered's git, ArduinoJson 7.4. Builds clean (78.8 % flash).
- Wi-Fi: hardcoded for v1 — credentials live in gitignored `secrets.h`.

### Decisions / context worth keeping
- PDM is the user's Python tooling default — saved as feedback memory.
- The Inkplate library 11.0 exposes web-image methods via
  `display.image.draw(url, x, y, dither, invert)`, not as direct methods
  on `Inkplate`. The first compile attempts using `drawImage` /
  `drawPngFromWeb` failed because those live on the inner `Image` member.
- `board_build.arduino.memory_type = qio_opi` is for ESP32-S3 only; using
  it on plain ESP32 caused `sdkconfig.h not found` build failures. Removed.
- Initial venv arch mismatch: PDM picked the system python.org x86_64 build
  but installed arm64 wheels. Forced `/opt/homebrew/bin/python3` (3.14, native
  arm64) and reinstalled — clean.

### Mid-day pivot — server becomes a dumb cache
After the first end-to-end run worked but every wake re-rendered, we had two
realizations and pivoted:

1. The server-side timestamp in the rendered image meant every poll produced
   a new content hash, defeating the whole "skip refresh on no change" idea.
   Briefly fixed by removing the timestamp; deeper issue: rendering on the
   server was conceptually wrong for what we actually want.
2. The renderer should live wherever the data lives (probably wholly external).
   The server's job collapses to: **store the latest PNG, serve it under
   bearer auth, hand out `last_modified`**. Push when something changes.

Refactored the server to:
- `POST /image` (PUSH_TOKEN) with optional `?dither=floyd-steinberg` to
  quantize onto the panel's 8 grays before storing
- `GET /config.json` + `/dashboard.png` (READ_TOKEN)
- `DELETE /image` (PUSH_TOKEN) for testing
- File-backed storage at `/data/dashboard.png` + `/data/meta.json`,
  atomic writes via tempfile+rename
- Bearer tokens via env, separate push and read so a leaked board token
  can't write

Firmware updated: `Authorization: Bearer READ_TOKEN` on both calls; the
image fetch now uses `HTTPClient` with the header set, then
`drawPngFromWeb(client*, ...)` consumes the stream directly.

### Deploy artifacts
- `server/Dockerfile` — multi-stage, exports PDM lockfile to pip-installable
  requirements so we don't fight PDM's nested venv layout in containers.
  Final image runs as uid 10001, read-only rootfs, `/data` volume.
- `deploy/k8s/{namespace,pvc,deployment,service,ingress,secret.example,
  kustomization}.yaml`. Single replica + Recreate strategy because the
  PVC is RWO and there's only one canonical image. Ingress terminates TLS
  (cluster decides cert-manager flavour).

### Decisions worth remembering for tomorrow
- The refactor settled the architecture question we kept circling: layout
  decisions live entirely outside this repo. This codebase only ever has
  to know about bytes.
- Pre-quantizing in the server is opt-in (`?dither=` query param). Default
  is passthrough so a smart pusher can do better dithering than us.
- Single-replica is a deliberate constraint — moving to HA needs RWX
  storage or external blob (S3 / object store), but that's a problem for
  when there are multiple displays, not a single Inkplate.
