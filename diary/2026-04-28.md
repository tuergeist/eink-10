## Channels — multi-board architecture

Picked Channel-based design over two separate services. All paths now
under `/c/{channel}/...` (legacy root paths dropped). Each board has
one channel; storage lives at `<DATA_DIR>/<channel>/`. Channel names
are validated against `^[a-z0-9][a-z0-9_-]{0,31}$` so user-supplied
strings can't escape the FS. Known channels (`inkplate10`, `ink6`)
also get a `panel` block in their `config.json` with `width`, `height`,
`gray_levels` so renderers can self-discover dimensions; ad-hoc names
work but skip the panel block.

Firmware took a `EINK_CHANNEL` compile-time macro driven by per-env
`build_flags` in `platformio.ini`. Two envs now: `inkplate10` and
`ink6`, both share `src/main.cpp`. The default config URL is built
from the macro so a single source line covers both boards.

`eink-push` CLI gained `--channel` (defaults to `inkplate10`).

## Inkplate 6 recovery saga

User said "kleinere Inkplate angeschlossen", I built + flashed the
ink6 firmware before they confirmed there was nothing to save on the
device. Mistake. They then said "backup the python script from the
device" — too late on the app partition (1 MB overwritten with our
Arduino firmware) but the data partition was untouched.

Rescue plan worked:
- esptool read-flash dumped the full 4 MB
- Inspecting the dump, the magic at 0x200000 is `littlefs` (not SPIFFS
  as the partition table claimed — that subtype 0x82 was misleading)
- mklittlefs from the PIO toolchain extracted 11 files: boot.py,
  main.py, sauna.py, wifi.py, image.py, inkplate6.py, gfx.py,
  gfx_standard_font_01.py, shapes.py, mcp23017.py, lib/urequests.py
- All saved under `backups/inkplate6_recovered/` (gitignored — likely
  contains Wi-Fi creds)

User then said "platt machen, software wie auf das andere display,
fixed to channel ink6". Did `esptool erase-flash` (full wipe, both
partitions), then `pio run -e ink6 -t upload`. Inkplate 6 now talks
the same protocol as the Inkplate 10, just at a different channel.

### Lessons
- "Nichts zu retten" was said for the first device on day 1. Don't
  assume that applies to every subsequent device. Always ask before
  destructive flash operations on hardware that's been in someone's
  hands.
- Partition table subtype labels lie. The Inkplate-MicroPython firmware
  ships with subtype=spiffs (0x82) but writes a LittleFS volume to it.
  Look at the magic bytes, not the table.
- mklittlefs in the PIO toolchain emits "Unsupported name_max" warnings
  but still works. Don't panic.

## Power-status overlay iterations

Three rounds today on the bottom-right battery indicator that landed
yesterday:

1. **Forced visual test.** Bumped `BATT_LOW` to `4.5f` so the threshold
   would fire on a charged cell, confirmed the `"LOW BAT 4.27V"` text
   overlay rendered correctly during a real refresh on the prod server.
2. **Switched to symbols.** Replaced the text with two hand-drawn
   icons via Adafruit GFX primitives — a battery outline with a small
   charge-stripe for `Low`, a USB trident for `Usb`. Both sit on a
   solid white box (color 7 in `INKPLATE_3BIT` mode) so they remain
   readable on dithered backgrounds. Hit a fun macro collision: the
   Arduino headers `#define LOW 0x0` and `HIGH 0x1`, and the
   preprocessor expands those even inside `enum class PowerState { OK,
   LOW, USB }` member names — had to rename to `Ok`/`Low`/`Usb`.
3. **Dropped the USB icon entirely.** User pushed back hard on the
   `EINK_HAS_BATTERY=0` build flag I'd added to "force USB classify
   when no cell is wired" — the flag is meaningless because (a) USB
   removal kills the board anyway and (b) plugging a battery in later
   wouldn't update the flag. The honest answer is that
   `display.readBattery()` on Inkplate 10 reads the charger IC's BAT
   pin, which floats to ~USB voltage when no cell is present. There's
   no exposed VBUS GPIO and no I2C charge-status register the library
   surfaces. So we can't reliably distinguish "USB-only, no battery"
   from "battery on USB charge". Just dropped the `Usb` state, kept
   `Ok` and `Low`, with `BATT_NOISE_FLOOR = 2.5V` as a sanity bound
   below which we assume a sensor noise / disconnected-divider
   scenario rather than a dying cell. Consumer-electronics convention:
   only show low-battery warnings, never bother showing "I'm plugged
   in".

### What the Inkplate 10 actually shows now
On a USB-powered unit (no battery), `readBattery()` returns ~4.27V →
state=OK → no overlay. Clock continues to render bottom-right when
`overlay_clock=true` from the server. Sleep stays at the configured
`refresh_interval_seconds`.

If a battery is later attached and drops below 3.4V, the low-battery
icon will appear top-right on the next image refresh, and sleep gets
multiplied by 6× to stretch remaining capacity.
